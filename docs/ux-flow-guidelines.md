# IEC 62443 协同工作台页面引导与入口规则

## 目标
降低首次使用门槛，让用户在每个主页面都能立即理解：
- 这页是做什么的
- 该由谁来填写或使用
- 完成后会进入哪一步

## 全局规则

### 1. 每个主页面顶部都应有任务导引区
统一回答三件事：
- 页面目的：当前页面要完成什么任务
- 适用角色：该由谁主导填写或维护
- 使用方式：用户在本页应如何开始，以及完成后会流向哪里

### 2. 主按钮只服务当前情境
- 首次进入：显示“开始当前任务”
- 中途返回：显示“继续当前任务”
- 已完成阶段：显示“查看当前结果”
- 不属于当前阶段的跳转按钮降级或隐藏

### 3. 页面文案以任务语言组织
避免只写功能名称，改为直接描述用户要完成的动作。

### 4. 页面之间保持阶段感
在主页面顶部提示：
- 当前处于哪一阶段
- 上一步是什么
- 下一步会进入哪里

## 页面建议

### 项目页
- 页面目的：建立项目共同上下文
- 适用角色：项目负责人 / 业主侧牵头人
- 使用方式：先填写项目基础信息，再进入业主需求整理
- 主按钮策略：
  - 未完成业主需求时：开始需求整理
  - 已有业主结果但后续未完成：继续当前步骤
  - 全流程已有结果时：查看当前结果

### 业主页
- 页面目的：整理业务目标、关键资产、风险关注点与目标等级输入
- 适用角色：业主 / 需求提出方
- 使用方式：完成访谈后进入结果页，供后续设计引用

### 集成商页
- 页面目的：将需求输入转为系统设计与约束方案
- 适用角色：集成商 / 设计负责人
- 使用方式：基于业主输入完成方案，再进入结果确认

### 设备商页
- 页面目的：声明产品能力、适配边界与限制条件
- 适用角色：设备商 / 产品或技术支持人员
- 使用方式：对照需求声明能力，完成后参与差距分析

### 差距页
- 页面目的：查看匹配结果、差距重点与补偿方向
- 适用角色：项目三方共同查看
- 使用方式：在已完成设计与能力声明后进入
